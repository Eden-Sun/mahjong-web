# 麻將牌辨識功能 — 實作計畫

> 目標：透過手機相機拍攝麻將牌面，自動辨識牌型並匯入遊戲系統。

## 名詞定義

| 縮寫 | 說明 |
|------|------|
| **S** | 小型工作量（1–2 天） |
| **M** | 中型工作量（3–5 天） |
| **L** | 大型工作量（1–2 週） |
| **P0** | 必須完成，阻擋上線 |
| **P1** | 重要，但可延後一個迭代 |
| **P2** | 錦上添花，資源充足再做 |

## 詳細任務

### 1. 技術研究與可行性驗證

| 任務 | 工時 | 優先級 | 說明 |
|------|------|--------|------|
| 調研現有麻將牌辨識方案（學術論文、開源專案） | S | P0 | 重點看 YOLOv8/v9、EfficientDet 在麻將牌上的表現 |
| 評估 on-device vs cloud 推論的延遲與成本 | S | P0 | 分別用 TensorFlow.js / ONNX Runtime Web 與 Cloud Run 測試 |
| 製作 POC：拍一張照片 → 輸出牌型陣列 | M | P0 | 端到端跑通，不要求精度，只要求管線完整 |
| 確定牌型編碼格式（對齊現有遊戲資料結構） | S | P0 | 與現有 `Tile` 型別對接 |

### 2. 資料收集與標註

| 任務 | 工時 | 優先級 | 說明 |
|------|------|--------|------|
| 收集麻將牌照片（不同光線、角度、背景） | L | P0 | 目標 5000+ 張，涵蓋台灣麻將 144 張牌所有花色 |
| 資料增強（旋轉、色偏、模糊、裁切） | M | P1 | 用 Albumentations 自動化 |
| 標註工具選型與標註流程建立 | S | P0 | 推薦 Label Studio 或 Roboflow |
| 執行標註（bounding box + 牌型分類） | L | P0 | 物件偵測標註，42 類（萬筒條字花） |
| 資料集切分（train/val/test = 70/15/15） | S | P1 | 確保各類別分布均勻 |

### 3. 模型選型與訓練

| 任務 | 工時 | 優先級 | 說明 |
|------|------|--------|------|
| **方案 A：On-device（優先）** | | | |
| 選定輕量模型架構（YOLOv8n / MobileNet-SSD） | S | P0 | 目標模型 < 10MB，推論 < 300ms |
| 訓練與調參 | L | P0 | GPU 環境用 Colab Pro 或自有機器 |
| 匯出至 TensorFlow.js / ONNX 格式 | M | P0 | 量化至 INT8 降低體積 |
| 在目標裝置（iOS Safari / Android Chrome）效能測試 | M | P0 | 低階手機也要能跑 |
| **方案 B：Cloud 推論（備案）** | | | |
| 部署模型至 Cloud Run / Lambda | M | P1 | 冷啟動 < 3s，推論 < 500ms |
| 設計圖片上傳 API（壓縮、resize） | S | P1 | 客戶端先壓至 640px 再上傳 |

### 4. API 設計

| 任務 | 工時 | 優先級 | 說明 |
|------|------|--------|------|
| 定義辨識結果 API schema | S | P0 | `POST /api/recognize` → `{ tiles: Tile[], confidence: number[], bbox: Rect[] }` |
| 定義校正回傳 API（使用者修正結果回饋） | S | P1 | 用於後續模型改進的標註資料 |
| API 限流與驗證機制 | S | P0 | 防濫用，rate limit per user |

```typescript
// 辨識結果型別定義
interface RecognitionResult {
  tiles: {
    id: string           // 牌型編碼，如 "1m", "9p", "east"
    confidence: number   // 0–1
    bbox: [x: number, y: number, w: number, h: number]
  }[]
  imageHash: string      // 用於去重
  processingMs: number   // 推論耗時
}
```

### 5. 前端相機 UX

| 任務 | 工時 | 優先級 | 說明 |
|------|------|--------|------|
| 相機權限請求與降級處理 | S | P0 | `getUserMedia` API，不支援時提供檔案上傳 |
| 相機取景框 UI（引導使用者對齊牌面） | M | P0 | 半透明遮罩 + 對齊框線 |
| 拍照 / 即時串流模式切換 | M | P1 | V1 先做拍照模式，V2 再做即時辨識 |
| 圖片預處理（裁切、壓縮、正規化） | S | P0 | Canvas API 處理 |
| 手電筒控制（低光環境） | S | P2 | `MediaStreamTrack.applyConstraints({ torch: true })` |
| 多張連拍辨識 | M | P2 | 拍整手牌可能需要多角度 |

### 6. 辨識管線整合

| 任務 | 工時 | 優先級 | 說明 |
|------|------|--------|------|
| 圖片前處理 pipeline（resize → normalize → tensor） | S | P0 | 統一輸入格式 |
| 模型載入與快取策略 | S | P0 | Service Worker 快取模型檔，避免重複下載 |
| 推論結果後處理（NMS、confidence 過濾） | M | P0 | 過濾低信心度結果，合併重疊框 |
| 牌型排序與分組邏輯 | S | P0 | 依 bbox x 座標排序，對應實際牌序 |
| On-device ↔ Cloud 自動切換 | M | P1 | 低階裝置自動 fallback 至 cloud |

### 7. 驗證與校正 UI

| 任務 | 工時 | 優先級 | 說明 |
|------|------|--------|------|
| 辨識結果預覽（原圖疊加 bounding box） | M | P0 | 使用者一眼確認是否正確 |
| 單張牌點擊修正（下拉選牌型） | M | P0 | 錯誤辨識可手動改 |
| 一鍵確認 / 全部重拍 | S | P0 | 確認後匯入遊戲 |
| 校正資料自動回傳（opt-in） | S | P1 | 使用者同意後回傳修正資料供再訓練 |

### 8. 測試

| 任務 | 工時 | 優先級 | 說明 |
|------|------|--------|------|
| 單元測試：牌型編碼轉換、排序邏輯 | S | P0 | |
| 整合測試：圖片 → 辨識 → 結果 pipeline | M | P0 | 用固定測試圖片集 |
| 辨識精度基準測試（target: mAP > 0.90） | M | P0 | 在 test set 上跑 |
| 跨裝置相容性測試 | M | P0 | iOS Safari、Android Chrome、主流機型 |
| 效能測試（記憶體、電量、發熱） | S | P1 | 長時間使用不能把手機搞當 |
| 使用者測試（5–10 人） | M | P1 | 真實場景回饋 |

## 技術風險與備案

| 風險 | 可能性 | 影響 | 備案 |
|------|--------|------|------|
| On-device 推論在低階手機太慢（> 1s） | 高 | 中 | Fallback 至 Cloud 推論；或降低模型輸入解析度 |
| 辨識精度不足（mAP < 0.85） | 中 | 高 | 增加訓練資料；嘗試更大模型（犧牲速度）；強化校正 UI 降低使用者痛感 |
| 特定光線 / 牌組辨識失敗率高 | 高 | 中 | 引導使用者調整光線；加入手電筒功能；資料增強覆蓋更多光線情境 |
| 瀏覽器相機 API 相容性問題 | 中 | 中 | 提供檔案上傳作為降級方案 |
| 模型檔太大影響首次載入 | 中 | 低 | 量化 + 分片載入 + Service Worker 快取 |
| 不同地區麻將牌面設計差異 | 低 | 中 | V1 先支援台灣常見牌面，後續擴充 |

## 建議執行順序

```
Phase 1（核心驗證）— 3–4 週
  ├── 技術研究 + POC
  ├── 資料收集與標註（同步進行）
  └── 模型初版訓練

Phase 2（產品化）— 3–4 週
  ├── 前端相機 UX
  ├── 辨識管線整合
  ├── 驗證校正 UI
  └── API 開發

Phase 3（上線）— 1–2 週
  ├── 測試與修正
  ├── 隱私安全確認
  ├── 灰度發布
  └── 監控建立
```

---

## 關鍵決策點

1. **POC 完成後**：辨識率是否達到可用門檻（> 80%）？不達標則重新評估方案。
2. **模型選型後**：on-device 能否在中階手機跑到 < 500ms？不行就改 cloud。
3. **Beta 測試後**：使用者修正頻率是否 < 20%？太高代表體驗不合格。
